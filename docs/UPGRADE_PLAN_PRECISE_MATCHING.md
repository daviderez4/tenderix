# תכנית שדרוג: התאמה מדויקת פרופיל חברה מול תנאי סף

**תאריך:** 17 פברואר 2026
**גרסה:** 1.0
**מטרה:** להגיע לרמת איכות שבה המערכת יודעת לבדוק בדיוק האם פרופיל חברה עומד בתנאי סף של מכרז - כולל הבנת הגדרות פנימיות של המכרז, זיהוי חוסרים, והסבר מפורט

---

## חזון מוביל הפרויקט

> "צריך לקחת מכרז ולכתוב פרופיל חברה שעונה במדויק על תנאי הסף, ואז המערכת תראה שהוא עומד ותסביר. אחר כך להזין פרופיל חברה שלא עומד בתנאי הסף ולנסות להכשיל את המערכת - כלומר אם דורשים במכרז פרויקט תחבורתי, לשים בפרופיל 3 פרויקטים תחבורתיים (לפי ההגדרה שלהם במכרז) ועוד 2 פרויקטים של בינוי, ולראות אם המערכת יודעת לעלות על זה ולהסביר מה ניתן לעשות"

---

## מצב קיים - מה עובד ומה לא

### ✅ מה עובד היום
| רכיב | סטטוס | הערות |
|-------|--------|-------|
| חילוץ תנאי סף מטקסט | עובד | Regex + AI |

### ❌ מה חסר - הבעיות המרכזיות

| בעיה | חומרה | הסבר |

### 🔴 שלב 1: חילוץ הגדרות המכרז (Definition-Aware Extraction)
**עדיפות:** קריטי | **מורכבות:** גבוהה | **זמן משוער:** Sprint 1

#### מה צריך לבנות:
המערכת חייבת להבין את ה"שפה" של כל מכרז. כשמכרז מגדיר "פרויקט תחבורתי" = פרויקט בתחום תחבורה ציבורית בהיקף של לפחות 10 מיליון ₪ - המערכת חייבת לזכור את זה ולהשתמש בזה.

#### משימות:

**1.1 - חילוץ אוטומטי של סעיף הגדרות (מודול 1.4)**
- סריקת מסמכי המכרז לאיתור סעיף "הגדרות" / "פרשנות מונחים"
- חילוץ כל זוג מונח → הגדרה → מגבלות
- שמירה בטבלת `tender_definitions` (קיימת בסכמה, לא מאוכלסת)
- דוגמה:
  ```
  מונח: "פרויקט דומה"
  הגדרה: "פרויקט בתחום מערכות תחבורה ציבורית בהיקף של לפחות 10 מיליון ₪ שבוצע ב-5 השנים האחרונות"
  מגבלות: היקף >= 10M, תחום = תחבורה ציבורית, שנים = 5
  ```

**1.2 - קישור הגדרות לתנאי סף**
- כל `gate_condition` יקושר ל-`tender_definition` הרלוונטי
- הוספת שדה `linked_definitions UUID[]` לטבלת `gate_conditions`
- כשמנתחים תנאי סף - המערכת מחפשת אם יש הגדרה במכרז למונח שמופיע בתנאי

**1.3 - Fallback להגדרות ממכרזים דומים**
- אם אין הגדרה במכרז הנוכחי → חיפוש במכרזים קודמים של אותו מזמין
- התראה למשתמש: "לא נמצאה הגדרה ל-X במכרז, נמצאה הגדרה דומה ממכרז Y"

#### קבצים שיושפעו:
- `database/schema/` - הוספת שדות ל-gate_conditions
- `mcp-servers/gate-extractor/index.js` - לוגיקת חילוץ הגדרות
- `ai/src/prompts/extraction.ts` - פרומפטים לחילוץ
- `frontend/src/pages/GatesPage.tsx` - הצגת הגדרות מקושרות
- `n8n/workflows/` - עדכון workflow של חילוץ

#### קריטריון הצלחה:
> כשהמערכת רואה תנאי סף "על המציע להציג 3 פרויקטים דומים" - היא יודעת להסתכל בהגדרות ולהבין ש"דומה" = תחבורה ציבורית ≥ 10M

### 🔴 שלב 2: מנוע התאמה סמנטי (Semantic Gate Matching Engine)
**עדיפות:** קריטי | **מורכבות:** גבוהה מאוד | **זמן משוער:** Sprint 2-3

#### מה צריך לבנות:
להחליף את מנוע ההתאמה מ-keyword-based ל-semantic + definition-aware.

#### משימות:

**2.1 - שכתוב לוגיקת ההתאמה ב-n8n**
```
הזרימה החדשה:
┌─────────────────────────────────────────────────────┐
│ 1. טען תנאי סף + הגדרות המכרז (tender_definitions) │
│ 2. טען פרופיל חברה (פרויקטים, פיננסי, הסמכות, כ"א) │
│ 3. לכל תנאי סף:                                     │
│    a. זהה את ההגדרות הרלוונטיות                      │
│    b. פרק לקריטריונים מדידים:                        │
│       - סוג פרויקט (לפי הגדרת המכרז!)               │
│       - היקף כספי                                     │
│       - טווח זמן                                      │
│       - ישות (מציע/ק.משנה)                            │
│    c. סרוק כל פרויקט בפרופיל:                        │
│       - האם מתאים לסוג? (לפי ההגדרה, לא keyword!)    │
│       - האם עומד בהיקף?                               │
│       - האם בטווח הזמן הנדרש?                         │
│    d. קבע סטטוס + ראיות + הסבר                       │
└─────────────────────────────────────────────────────┘
```

**2.2 - AI-Powered Project Classification**
- לכל פרויקט בפרופיל - שאל את Claude: "האם פרויקט X מתאים להגדרה Y?"
- התבסס על ההגדרה המדויקת מהמכרז, לא על keywords
- דוגמה קריטית:
  ```
  הגדרת המכרז: "פרויקט תחבורתי" = פרויקט בתחום תחבורה ציבורית

  פרויקט "בניית כביש גישה" → בינוי, לא תחבורה ציבורית → ❌ לא עומד
  פרויקט "מערכת ניהול רמזורים" → תחבורה ציבורית → ✅ עומד
  פרויקט "פיתוח אפליקציית שילוט דיגיטלי" → לא תחבורה → ❌ לא עומד
  ```

**2.3 - Matching Confidence & Reasoning**
- לכל התאמה: `confidence_score` (0-100)
- שרשרת הסבר (reasoning chain):
  ```json
  {
    "condition": "3 פרויקטים תחבורתיים בהיקף 10M+ ב-5 שנים אחרונות",
    "definition_used": "פרויקט תחבורתי = תחבורה ציבורית",
    "projects_evaluated": [
      {
        "name": "מערכת ניהול רמזורים",
        "classification": "תחבורה ציבורית ✅",
        "value": "15M ✅",
        "date_range": "2022-2024 ✅",
        "reasoning": "מערכת ניהול רמזורים היא חלק מתשתית תחבורה ציבורית"
      },
      {
        "name": "בניית כביש גישה",
        "classification": "בינוי ❌",
        "reasoning": "כביש גישה הוא פרויקט בינוי תשתיות, לא תחבורה ציבורית לפי ההגדרה"
      }
    ],
    "matching_count": 2,
    "required_count": 3,
    "status": "DOES_NOT_MEET",
    "gap": "חסר פרויקט תחבורתי אחד. יש 2 מתוך 3 נדרשים.",
    "suggestions": ["חיפוש שותף/ק.משנה עם ניסיון בתחבורה ציבורית"]
  }
  ```

**2.4 - אנטי-מניפולציה (Adversarial Detection)**
- זיהוי פרויקטים שנראים מתאימים אבל לא באמת:
  - פרויקט עם מילת מפתח בשם אבל לא בתוכן
  - פרויקט שהגדרתו עמומה
  - פרויקט שנספר פעמיים (C3)
- דגלים אדומים:
  ```
  ⚠️ "פרויקט 'שדרוג תשתיות תחבורה' - שם הפרויקט מכיל 'תחבורה' אך
  מתיאור הפרויקט עולה שמדובר בשדרוג תשתיות תקשורת לחברת תחבורה,
  ולא בפרויקט תחבורתי לפי הגדרת המכרז"
  ```

#### קבצים שיושפעו:
- `n8n/workflows/TDX-P2-Gate-Matching-FINAL.json` - שכתוב מנוע ההתאמה
- `mcp-servers/gate-analyzer/index.js` - הוספת AI classification
- `ai/src/prompts/analysis.ts` - פרומפטים להתאמה סמנטית
- `frontend/src/api/tenderix.ts` - עדכון API calls
- `database/schema/` - הרחבת gate_condition_matches

#### קריטריון הצלחה:
> פרופיל עם 3 פרויקטים תחבורתיים + 2 בינוי → המערכת מזהה רק 3 כתחבורתיים, לא 5.
> פרופיל עם 2 תחבורתיים + 1 "כביש" → המערכת אומרת "2 מתוך 3, כביש = בינוי לא תחבורה"

### 🟡 שלב 3: הסברים מפורטים למשתמש (Explainability Layer)
**עדיפות:** גבוהה | **מורכבות:** בינונית | **זמן משוער:** Sprint 3

#### מה צריך לבנות:
כל החלטה של המערכת חייבת להיות מוסברת בעברית ברורה - למה עומד, למה לא עומד, מה אפשר לעשות.

#### משימות:

**3.1 - Explanation Template Engine**
```
לכל תנאי סף, הצג:
┌──────────────────────────────────────────────────┐
│ תנאי סף #3: 3 פרויקטים תחבורתיים ≥ 10M ₪       │
│ סטטוס: ❌ לא עומד (2 מתוך 3)                     │
│                                                    │
│ 📋 הגדרת המכרז:                                   │
│ "פרויקט תחבורתי" = פרויקט בתחום תחבורה ציבורית   │
│ בהיקף 10 מיליון ₪ לפחות, שבוצע ב-5 שנים אחרונות │
│                                                    │
│ ✅ פרויקטים שעומדים (2):                           │
│ 1. מערכת ניהול רמזורים - 15M - 2022              │
│    → תחבורה ציבורית, עומד בהיקף ובתאריך          │
│ 2. מערכת כרטוס חכם - 12M - 2023                  │
│    → תחבורה ציבורית, עומד בהיקף ובתאריך          │
│                                                    │
│ ❌ פרויקטים שלא עומדים (3):                        │
│ 3. בניית כביש גישה - 20M - 2023                  │
│    → בינוי, לא תחבורה ציבורית לפי ההגדרה         │
│ 4. שדרוג מערכת תאורה - 8M - 2024                 │
│    → עומד בתחום אך לא בהיקף (8M < 10M)           │
│ 5. פיתוח אפליקציית ניווט - 5M - 2024             │
│    → תוכנה, לא תחבורה ציבורית                     │
│                                                    │
│ 💡 מה אפשר לעשות:                                 │
│ • חיפוש שותף עם פרויקט תחבורתי נוסף              │
│ • בדיקה אם "כביש גישה" ניתן לפרשנות מרחיבה      │
│ • שאלת הבהרה: האם פרויקט תאורה ≥ 8M נחשב?       │
└──────────────────────────────────────────────────┘
```

**3.2 - UI Component: GateMatchExplanation**
- קומפוננטה חדשה שמציגה את ההסבר המלא
- Expandable - סגור בברירת מחדל, נפתח בלחיצה
- צבעים: ירוק לעומד, אדום ללא עומד, כתום לחלקי
- אייקונים לסוג הפער

**3.3 - דוח PDF מסכם**
- הפקת דוח PDF/HTML עם כל ההסברים
- מיועד להצגה בפני הנהלת החברה

#### קבצים שיושפעו:

- `frontend/src/pages/GatesPage.tsx` - שילוב ההסברים

#### קריטריון הצלחה:
> המשתמש רואה הסבר מלא לכל תנאי סף - כולל ציטוט ההגדרה מהמכרז, פירוט כל פרויקט, ולמה כל אחד עומד/לא עומד

### 🟡 שלב 4: יצירת פרופיל אוטומטית ממכרז (Profile Generator)
**עדיפות:** גבוהה | **מורכבות:** בינונית | **זמן משוער:** Sprint 4

#### מה צריך לבנות:
כלי שלוקח מכרז ויוצר אוטומטית פרופיל חברה לדוגמה - פרופיל שעומד בדיוק בתנאי הסף, ופרופיל שלא עומד.

#### משימות:

**4.1 - Profile Generator: "פרופיל עובר"**
- קח את כל תנאי הסף של המכרז
- צור פרויקטים, הסמכות, אנשי מפתח ופיננסים שעונים בדיוק
- השתמש בהגדרות המכרז ליצירת פרויקטים ריאליסטיים
- דוגמה:
  ```
  תנאי: 3 פרויקטים תחבורתיים ≥ 10M
  → יוצר 3 פרויקטים: "מערכת ניהול רמזורים" (15M),
    "כרטוס חכם" (12M), "בקרת תנועה" (11M)
  ```

**4.2 - Profile Generator: "פרופיל נכשל (אדוורסריאלי)"**
- יוצר פרופיל שנראה תואם אבל לא באמת
- שמות פרויקטים מטעים (מכילים מילות מפתח אבל מתחום אחר)
- מספרים קרובים אך לא מספיקים
- פרויקטים מהתחום הנכון אבל מחוץ לטווח הזמן
- דוגמה:
  ```
  תנאי: 3 פרויקטים תחבורתיים ≥ 10M
  → יוצר: "שדרוג תשתיות תחבורה" (בינוי, 20M) - מטעה!
    "מערכת רמזורים" (8M) - מתחת להיקף!
    "כרטוס חכם" (12M, 2018) - מחוץ לטווח!
  ```

**4.3 - כפתור "בדוק את המערכת" (Self-Test)**
- ממשק שמאפשר ליצור פרופיל-דוגמה ולהריץ matching
- מציג side-by-side: פרופיל עובר vs פרופיל נכשל
- מאמת שהמערכת מזהה נכון את שניהם

#### קבצים חדשים:
- `frontend/src/pages/ProfileTestPage.tsx` - דף בדיקת פרופילים
- `ai/src/agents/profile-generator.ts` - סוכן יצירת פרופילים
- `ai/src/prompts/profile-generation.ts` - פרומפטים ליצירה

#### קריטריון הצלחה:
> ניתן ללחוץ כפתור ולקבל פרופיל-דוגמה שעומד + פרופיל שלא עומד, והמערכת מזהה נכון את שניהם עם הסברים

### 🟢 שלב 5: מילון טכני ופרשנות (Technical Dictionary & Interpretation)
**עדיפות:** בינונית-גבוהה | **מורכבות:** גבוהה | **זמן משוער:** Sprint 5

#### משימות:

**5.1 - בניית מילון טכני (Core C2)**
- אכלוס טבלת `technical_dictionary` עם מונחים מתחומים שונים
- מיפוי "שווה ערך": תחבורה ציבורית = כולל רכבת קלה, אוטובוסים, מטרו
- מיפוי "לא שווה ערך": תחבורה ציבורית ≠ כביש, מנהרה, גשר
- עדכון דינמי מכל מכרז חדש

**5.2 - מנגנון פרשנות כפול (מודול 2.5)**
```
"ראש משפטי" (פרשנות מצמצמת):
  → מה כתוב בדיוק? אם לא מוגדר - לא מתאים
  → "כביש גישה" ≠ "תחבורה ציבורית" כי לא מוגדר כך

"ראש טכני" (פרשנות מרחיבה):
  → מה הכוונה? אם דומה - אולי מתאים

```

- המלצה: "לפי פרשנות מצמצמת - לא עומד. לפי פרשנות מרחיבה - ייתכן. מומלץ לשאול הבהרה"

## סדר עדיפויות - מה קודם למה

```
שלב 1 (הגדרות) ────────► שלב 2 (מנוע סמנטי) ────► שלב 3 (הסברים)
                                    │
                                    ▼
                            שלב 4 (generator)
                                    │
שלב 5 (מילון + פרשנות) ◄──────────┘
                                    │
                                    ▼
                            שלב 6 (QA + בדיקות)
```

**שלבים 1 → 2 הם חובה ברצף** - אי אפשר לעשות התאמה סמנטית בלי הגדרות.
**שלב 3 יכול במקביל לשלב 4** - הסברים וגנרטור הם עצמאיים.
**שלב 5 משפר את שלבים 1-2** - המילון הטכני מעשיר את ההגדרות.
**שלב 6 בסוף** - בדיקות E2E אחרי שהכל בנוי.

## סיכום

התכנית מתמקדת ב-3 עקרונות מפתח:

1. **הגדרות המכרז קודמות להכל** - המערכת חייבת להבין את השפה הספציפית של כל מכרז
2. **הסבר = אמינות** - כל החלטה חייבת להיות מוסברת ומנומקת
3. **אנטי-מניפולציה** - המערכת חייבת לזהות פרופילים "מטעים" ולא להיתפס

בסיום 6 השלבים, המערכת תוכל:
- לקחת כל מכרז → להבין את ההגדרות שלו
- לבדוק כל פרופיל חברה → לפי ההגדרות המדויקות
- להסביר בדיוק למה עומד / למה לא
- לזהות פרופילים שנראים טובים אך לא באמת עומדים
- להציע דרכים לסגור פערים
